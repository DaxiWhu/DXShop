# MQ死信队列监控 - 5分钟快速开始

## 🚀 快速部署（5分钟）

### 第1步：执行建表SQL（1分钟）

```bash
mysql -h 172.17.156.101 -u root -p123456 dxshop < src/main/resources/sql/mq_dead_letter_message.sql
```

或者在MySQL客户端中执行：
```sql
-- 复制 sql/mq_dead_letter_message.sql 的内容并执行
```

### 第2步：配置告警（可选，1分钟）

编辑 `application.yaml`：

```yaml
mq:
  alert:
    enabled: true  # 启用告警
    dingtalk:
      webhook: https://oapi.dingtalk.com/robot/send?access_token=YOUR_TOKEN
```

**获取钉钉Token**：
1. 钉钉群 → 智能群助手 → 添加机器人 → 自定义
2. 安全设置：关键词 `DXShop`
3. 复制Webhook地址

**如果不配置钉钉**：系统仍会记录到数据库和日志，只是不发送钉钉通知。

### 第3步：重启应用（2分钟）

```bash
cd DXShop-order
mvn clean package -DskipTests
java -jar target/DXShop-order.jar
```

### 第4步：验证部署（1分钟）

#### 检查日志

```bash
tail -f ./logs/DXShop-order.log | grep "Dlq"
```

应该看到类似输出：
```

RocketMQListenerContainer : running container: DefaultRocketMQListenerContainer{consumerGroup='dlq-refund-timeout-consumer-group'...}
RocketMQListenerContainer : running container: DefaultRocketMQListenerContainer{consumerGroup='dlq-address-modify-timeout-consumer-group'...}
```

#### 测试接口

```bash
curl http://localhost:8080/api/mq/dead-letter/unprocessed-count
```

预期响应：


## 📱 如何使用

### 场景1：收到告警后处理

1. **收到钉钉告警**
   ```
   【DXShop】MQ死信消息告警
   
   业务类型: 订单超时取消
   消息ID: AC1400017B2E18B4AAC27F6A7C5E0000
   业务键: 123456789
   ...
   ```

2. **查看详情**
   ```bash
   curl http://localhost:8080/api/mq/dead-letter/1
   ```

3. **人工处理**
   - 登录系统检查订单状态
   - 手动取消订单或执行其他操作

4. **标记为已处理**
   ```bash
   curl -X POST "http://localhost:8080/api/mq/dead-letter/handle/1?handler=张三&remark=已手动取消订单"
   ```

### 场景2：日常巡检

```bash
# 查看未处理数量
curl http://localhost:8080/api/mq/dead-letter/unprocessed-count

# 如果有未处理的消息，查看详情
curl "http://localhost:8080/api/mq/dead-letter/page?pageNum=1&pageSize=10&status=0"
```

### 场景3：清理历史数据

```bash
# 批量删除已处理的消息（假设ID为1,2,3）
curl -X DELETE http://localhost:8080/api/mq/dead-letter/batch-delete \
  -H "Content-Type: application/json" \
  -d '[1, 2, 3]'
```

## 🔍 常用命令

### 查看数据库记录

```sql
-- 查看所有未处理的消息
SELECT * FROM mq_dead_letter_message WHERE status = 0 ORDER BY create_time DESC;

-- 按业务类型统计
SELECT business_type, COUNT(*) as count 
FROM mq_dead_letter_message 
WHERE status = 0 
GROUP BY business_type;
```

## 🆘 常见问题

### Q1: 如何知道部署是否成功？

**检查清单**：
- [ ] 数据库表 `mq_dead_letter_message` 存在
- [ ] 日志中有3个死信消费者的启动信息
- [ ] 访问 `/api/mq/dead-letter/unprocessed-count` 返回正常

### Q2: 没有收到钉钉告警怎么办？

**不影响使用**！即使没有配置钉钉，系统仍然会：
- ✅ 记录到数据库
- ✅ 记录到日志

你可以定期查询数据库或日志来发现死信消息。

### Q3: 如何测试功能是否正常？

**方法1**：插入测试数据
```sql
INSERT INTO mq_dead_letter_message (
    msg_id, topic, consumer_group, business_key, 
    retry_count, error_message, business_type, status
) VALUES (
    'TEST_001', 
    '%DLQ%order-timeout-consumer-group',
    'order-timeout-consumer-group',
    '999999',
    16,
    '测试消息',
    'ORDER_TIMEOUT',
    0
);
```

然后查询：
```bash
curl http://localhost:8080/api/mq/dead-letter/unprocessed-count
```

应该返回 `1`。

**方法2**：等待真实业务触发

### Q4: 死信消息会不会影响正常业务？

**不会**！死信消息是已经重试16次失败的消息，进入死信队列后：
- 不会影响正常消息的消费
- 死信消费者只负责记录和告警
- 不会再次投递到正常队列

### Q5: 需要定期清理数据吗？

**建议**：每月清理一次已处理的历史数据

```sql
-- 删除3个月前已处理的消息
DELETE FROM mq_dead_letter_message 
WHERE status IN (1, 2) 
  AND handle_time < DATE_SUB(NOW(), INTERVAL 3 MONTH);
```
