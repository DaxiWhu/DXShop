# MQ死信队列监控方案 - 实施总结

## 📋 方案概述

本方案实现了RocketMQ死信队列的完整监控和告警机制，解决了消息重试多次失败后无人知晓的问题。

### 核心功能

1. **自动监听**：3个死信消费者自动监听不同业务的死信队列
2. **持久化记录**：所有死信消息保存到数据库，便于追溯
3. **实时告警**：通过钉钉机器人发送告警通知
4. **管理接口**：提供完整的API用于查看和处理死信消息
5. **双重保障**：数据库记录 + 日志记录，确保不丢失任何告警

## 🏗️ 架构设计

### 整体流程

```
┌─────────────────────────────────────────────────────────────┐
│                     正常消费者处理消息                          │
│                  (OrderTimeoutConsumer等)                     │
└────────────────────┬────────────────────────────────────────┘
                     │ 处理失败，抛出异常
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                  RocketMQ自动重试机制                          │
│                   最多重试16次（约4.5小时）                     │
└────────────────────┬────────────────────────────────────────┘
                     │ 16次重试全部失败
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                    消息进入死信队列                             │
│         Topic: %DLQ% + 消费者组名                              │
└────────────────────┬────────────────────────────────────────┘
                     │ 
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                   死信消费者监听                               │
│         (DlqOrderTimeoutConsumer等)                           │
└────────────────────┬────────────────────────────────────────┘
                     │ 接收死信消息
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                  MqAlertService处理                           │
│          1. 记录到数据库 (mq_dead_letter_message)             │
│          2. 发送钉钉告警（如果配置）                            │
│          3. 记录错误日志                                      │
└─────────────────────────────────────────────────────────────┘
```

### 数据流转

```
死信消息 → 死信消费者 → MqAlertService → 数据库记录
                                        → 钉钉告警
                                        → 日志记录
                                              ↓
                                    管理员收到告警
                                              ↓
                                    调用管理接口处理
                                              ↓
                                    更新状态为"已处理"
```

## 🔧 核心代码说明

### 1. 死信消费者

每个死信消费者的实现非常简单：

```java
@RocketMQMessageListener(
    topic = "%DLQ%order-timeout-consumer-group",
    consumerGroup = "dlq-order-timeout-consumer-group"
)
public class DlqOrderTimeoutConsumer implements RocketMQListener<MessageExt> {
    
    private final MqAlertService mqAlertService;
    
    @Override
    public void onMessage(MessageExt message) {
        // 记录并告警
        mqAlertService.recordAndAlert(
            message, 
            "order-timeout-consumer-group",
            "ORDER_TIMEOUT",
            "订单超时处理失败，重试16次后进入死信队列，需要人工介入处理"
        );
        // 不抛出异常，避免无限循环
    }
}
```

**关键点**：
- 监听死信Topic（%DLQ% + 原消费者组名）
- 不抛出异常，避免死信消息无限循环
- 只负责记录和告警，不进行业务处理

### 2. 告警服务

```java
public void recordAndAlert(MessageExt messageExt, String consumerGroup, 
                          String businessType, String errorMessage) {
    // 1. 构建死信消息对象
    MqDeadLetterMessage deadLetterMessage = new MqDeadLetterMessage();
    // ... 设置各种属性
    
    // 2. 记录到数据库
    deadLetterMessageMapper.insert(deadLetterMessage);
    
    // 3. 发送告警
    if (alertEnabled) {
        sendAlertNotification(deadLetterMessage);
    }
}
```

**双重保障**：
- 数据库记录：永久保存，可追溯
- 日志记录：即使数据库失败也有日志

## 📈 监控指标

### 建议监控的指标

1. **未处理死信消息数量**
   - 正常值：0
   - 警告阈值：> 10
   - 严重阈值：> 50

2. **死信消息产生速率**
   - 正常值：0条/天
   - 警告阈值：> 5条/天
   - 严重阈值：> 20条/天

3. **平均处理时长**
   - 从产生到处理的时间
   - 目标：< 24小时

### 监控SQL

```sql
-- 未处理数量
SELECT COUNT(*) FROM mq_dead_letter_message WHERE status = 0;

-- 按业务类型统计
SELECT business_type, COUNT(*) as count 
FROM mq_dead_letter_message 
WHERE status = 0 
GROUP BY business_type;

-- 今日新增
SELECT COUNT(*) FROM mq_dead_letter_message 
WHERE create_time >= CURDATE();

-- 平均处理时长（小时）
SELECT AVG(TIMESTAMPDIFF(HOUR, create_time, handle_time)) 
FROM mq_dead_letter_message 
WHERE status = 1 AND handle_time IS NOT NULL;
```

## 🔍 故障排查

### 常见问题

#### 问题1：死信消费者没有启动

**症状**：日志中没有死信消费者的启动信息

**排查**：
1. 检查RocketMQ连接是否正常
2. 检查消费者组名称是否正确
3. 检查Topic是否存在

**解决**：
- 确认RocketMQ服务正常
- 检查配置中的NameServer地址
- 重启应用

#### 问题2：告警没有发送

**症状**：收到死信消息但没有钉钉通知

**排查**：
```bash
grep "钉钉告警" ./logs/DXShop-order.log
```

**可能原因**：
- Webhook配置错误
- 网络连接问题
- 钉钉机器人安全设置不匹配

**解决**：
- 检查Webhook地址是否正确
- 测试网络连通性
- 确认钉钉机器人关键词包含"DXShop"

#### 问题3：数据库记录失败

**症状**：日志显示"记录死信消息失败"

**排查**：
1. 检查数据库连接
2. 检查表是否存在
3. 检查字段是否匹配

**解决**：
- 重新执行建表SQL
- 检查数据库权限
- 查看具体错误信息
